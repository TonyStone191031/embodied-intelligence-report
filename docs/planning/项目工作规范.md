# 项目工作规范

## 1. 目的

本文件是本项目的长期操作手册，供后续在新会话中继续维护《具身智能行业和技术发展调研报告》的大模型直接阅读。

进入本仓库后，优先阅读：

1. 本文件
2. `docs/planning/具身智能报告-V0.0-详细大纲.md`

---

## 2. 核心规则

### 2.1 先大纲，后正文

任何版本更新都必须先做大纲，不允许直接写正文。

标准顺序：

1. 阅读本文件与现有报告材料
2. 在 `docs/planning/` 新建或更新该版本大纲
3. 等待用户确认大纲
4. 大纲确认后，才开始正文写作

### 2.2 增量更新，不推倒重写

新版本默认基于旧版本增量修订：

1. 新论文、新模型、新企业、新政策补充
2. 旧判断修正
3. 图表、代码、引用和案例复用

### 2.3 区分三类内容

报告内容尽量分清：

1. 事实
2. 推断
3. 评价

不得把推断或评价写成事实。

## 3. 目录职责

### 3.1 `docs/planning/`

放规划文件：

1. 工作规范
2. 各版本详细大纲
3. skills 调用备忘
4. 其他阶段性规划文档

### 3.2 `docs/report/`

放报告正文与版本目录：

1. `current/`：当前正在维护的工作版本
2. `versions/`：已冻结的历史版本

### 3.3 `research/`

放调研过程资料，不放正式正文：

1. 论文笔记
2. 企业笔记
3. 数据集与 benchmark 说明
4. 行业文章摘录
5. 时间线材料

### 3.4 `assets/`

放图片资产原件与可复用图示：

1. `images/`
2. `diagrams/`
3. `screenshots/`

### 3.5 `code/`

放抓取、分析、制图脚本。

### 3.6 `data/`

放原始数据与处理后数据。

### 3.7 `output/`

放导出产物，如合并版 markdown、pdf、docx、截图输出。

## 4. 大纲与正文流程

### 4.1 新版本启动时

先检查：

1. `CHANGELOG.md`
2. 上一版本正文
3. 上一版本大纲
4. 本文件

### 4.2 新大纲放哪里

统一放在：

`docs/planning/`

命名建议：

`具身智能报告-VX.Y-详细大纲.md`

### 4.3 什么时候开始写正文

只有在用户明确确认大纲后，才允许写正文。

### 4.4 正文如何组织

正文按章节拆分维护，不长期维护单个超大 markdown。

### 4.5 正文默认对齐已确认大纲，但允许“研究驱动重构”

本项目默认要求正文对齐已确认详细大纲，但该大纲不是不可更改的静态壳，而是会随着深度研究推进而持续校准的研究骨架。

基本原则：

1. 在未出现明显结构性问题前，`docs/report/current/` 下的章节文件应与当前有效大纲中的“部分”一一对应
2. 每个章节文件内部的小节顺序，应与当前有效大纲中的该部分顺序保持一致
3. 正文写作时可以在当前有效大纲之下继续下探更细标题
4. 如果在调用 skills、补充文献和完成章节级研究后，发现现有大纲会实质性限制正文质量、论证完整性或章节边界合理性，则允许触发局部或阶段性重构
5. 发生重构时，必须先回到 `docs/planning/` 更新详细大纲，再继续受影响章节的正文写作
6. 对于已写章节，如其职责已因重构发生变化，应同步回修其标题、导语、结构说明和交叉引用，避免“旧结构正文”挂在“新结构大纲”下
7. 入口文件、图表编号、表格编号、代码片段编号与交叉引用，应尽量跟随最新有效大纲同步维护

重构触发条件：

1. 章节级深度研究表明原大纲对问题切分过浅、过窄或存在明显错位
2. 新增材料显示多个章节边界划分不合理
3. 当前章节若继续沿原大纲写作，会明显损害报告的完整性、严谨性或长期可维护性

重构流程：

1. 先做章节级研究
2. 明确原结构的问题
3. 在 `docs/planning/` 更新详细大纲
4. 向用户同步说明调整理由
5. 再继续相关正文编写或回修

禁止事项：

1. 未更新大纲就直接长期沿另一套结构写正文
2. 研究已证明结构失真，但仍机械坚持旧大纲推进
3. 正文文件名与当前有效大纲长期不一致
4. 在正文、图表和引用中并行维护多套编号体系，导致后续版本无法维护

### 4.6 正文必须按“研究型章节流程”推进

本项目的正文不是科普稿，也不是先写概述、后补证据的随手草稿。每一章都应按“先研究、后写作、再审查”的研究型流程推进。

章节级标准流程：

1. 先明确本章目标
2. 先做章节级研究与资料收集
3. 再写正式正文
4. 再做章节级审查
5. 通过后才继续下一章

其中：

1. “先明确本章目标”包括明确本章要回答的问题、章节内部结构、关键争议、图表需求、引用需求
2. “章节级研究与资料收集”包括论文、综述、技术报告、项目主页、GitHub、官方博客、企业发布材料、benchmark 页面等
3. “正式正文”必须建立在已整理的研究材料上，不允许先凭印象写概述再长期拖延补证据
4. “章节级审查”必须检查论证链、引用密度、事实/推断/评价分层、章节覆盖范围和图表规划是否达标

禁止事项：

1. 未形成该章研究材料就直接大段写正文
2. 用少量常识性描述替代系统研究
3. 章节只有概念介绍，没有代表路线、工程约束、局限性和争议分析
4. 把“后面再补引用”作为常态工作方式

### 4.7 章节级最小交付标准

每一章在被视为“正文初稿”前，至少应尽量包含以下要素：

1. 本章核心问题与边界
2. 该方向的基本定义与背景
3. 代表性技术路线或分析框架
4. 关键论文、项目、系统或企业实例
5. 工程实现路径或现实部署约束
6. 局限性、争议点与未解决问题
7. 后续章节接口，即本章与后文的衔接关系

如果某章暂时做不到上述全部内容，必须明确标注缺口，而不是默认以简短概述稿充当正式正文。

### 4.8 章节级推荐工作流与强制质量门

本项目推荐并优先采用如下章节工作流：

1. 用 `writing-plans` 设计该章任务拆解
2. 用 `academic-research-suite` 做章节级深度研究
3. 结合 `pdf`、`playwright`、`screenshot`、`web` 补论文、官网、GitHub、图示与截图证据
4. 在 `research/` 中沉淀研究材料与笔记
5. 再写入 `docs/report/current/` 对应章节正文
6. 用 `paper-spine-audit` 的思路检查论证链与章节深度
7. 用 `verification-before-completion` 的思路做交付前自检

其中第 2、5、6 步对本项目的研究型章节写作而言应视为默认强制环节，而不是可选优化项。

### 4.9 章节写作中的“动态回调大纲”机制

本项目采用“先有整体大纲，再做章节级重估”的工作方式。

即：

1. 先调用适合的 skills 制定一版全书整体大纲
2. 在编写每一章前，继续调用适合的 skills 做该章研究
3. 如果该章研究表明原大纲章节边界不合理，则允许回调并优化整体大纲
4. 大纲更新后，再同步该章及相邻章节的职责与接口

因此，大纲确认并不意味着后续不能调整；它意味着后续调整必须有研究依据、有文档同步，并对已写内容负责。

### 4.10 低打扰批次推进机制

当用户明确授权后，长篇报告写作可切换为“低打扰批次推进”模式，而不再要求每完成一章都单独等待用户确认。

该模式下：

1. 仍然必须坚持“先研究、后写作、再审查”
2. 章节级结构仍允许依据研究结果回调大纲
3. 但沟通与汇报单位从“单章”提升为“批次”
4. 每个批次结束后，再统一向用户汇报结构变化、章节状态和后续计划

默认批次定义、上下文管理策略、图表 / 公式 / 代码片段策略，见：

`docs/planning/V0.0-低打扰自主推进方案.md`

## 6. 报告格式要求

1. 报告使用 Markdown
2. 标题带版本号，例如 `《具身智能行业和技术发展调研报告-v0.0》`
3. 文风偏正式、严谨
4. 关键观点尽量带来源说明
5. 关键架构、时间线、模型谱系、对比关系尽量配图或配表
6. 关键方法必要时配代码片段或伪代码

每个技术方向尽量回答：

1. 是什么
2. 为什么重要
3. 怎么做
4. 工程上怎么落地
5. 局限是什么
6. 未来可能怎么发展

## 7. 资料组织规则

### 7.1 研究资料与正文分离

1. 正文文件只写整理后的结论和论述
2. 原始调研笔记放 `research/`
3. 抓取脚本和分析脚本放 `code/`

### 7.2 图片放置规则

1. 可复用原图：`assets/images/`
2. 自绘图：`assets/diagrams/`
3. 网页/公告截图：`assets/screenshots/`
4. 某版本专用图：`docs/report/versions/<version>/figures/`

### 7.3 代码片段放置规则

1. 分析脚本：`code/analysis/`
2. 抓取脚本：`code/scrapers/`
3. 某版本专用片段：`docs/report/versions/<version>/snippets/`

### 7.4 表格放置规则

1. 某版本整理后表格：`docs/report/versions/<version>/tables/`
2. 底层数据：`data/raw/` 或 `data/processed/`

### 7.5 命名建议

1. 论文笔记：`年份-机构-标题缩写.md`
2. 企业笔记：`企业名-更新日期.md`
3. 时间线：`timeline-主题.md`
4. 截图：`日期-来源域名-主题.png`
5. 图表：`章节号-主题.ext`

## 8. 版本管理规则

### 8.1 不建议“每个版本一个 git 分支”

默认不用 git 分支来管理每个报告版本。

原因：

1. 分支适合并行开发和实验
2. 报告版本更适合做内容演进
3. 多分支会让图表、引用和章节历史变得分叉且难维护

### 8.2 采用“目录版本化 + Git 历史”

默认采用：

1. Git 主线维护整体历史
2. 报告版本通过文件与目录管理
3. 重大实验性改动才临时开分支

### 8.3 版本目录结构

```text
docs/report/
  current/
  versions/
    v0.0/
    v0.1/
```

含义：

1. `current/`：当前工作版本
2. `versions/vX.Y/`：已冻结版本

### 8.4 工作流

1. 平时在 `current/` 中工作
2. 某版本确认完成后，整理到 `versions/<version>/`
3. 新版本继续在 `current/` 上增量推进

### 8.5 避免覆盖旧版本

1. 已冻结版本目录默认不持续改写
2. 新版本不直接覆盖旧版本目录
3. 新版本必须新建自己的 `versions/<version>/`

### 8.6 版本资产管理

公共可复用资产继续放：

1. `assets/images/`
2. `assets/diagrams/`
3. `assets/screenshots/`
4. `code/analysis/`
5. `data/raw/`
6. `data/processed/`

版本专用资产冻结到：

1. `docs/report/versions/<version>/figures/`
2. `docs/report/versions/<version>/tables/`
3. `docs/report/versions/<version>/snippets/`

### 8.7 版本发布时同步更新

1. `CHANGELOG.md`
2. 对应版本目录
3. 版本入口文件
4. 版本专用图表与代码片段目录

## 9. 新版本标准流程

### 9.1 启动

1. 阅读本文件
2. 阅读上一版本正文
3. 阅读上一版本大纲
4. 阅读 `CHANGELOG.md`
5. 调研新增内容

### 9.2 规划

1. 新建该版本详细大纲
2. 标出新增、修订、删除或降级章节
3. 明确本版本重点增量
4. 等待用户确认

### 9.3 写作

1. 按章节推进
2. 先基础层，后技术层，再产业与结论层
3. 每章同步补图、表、引用和待核实项
4. 每章先完成研究材料，再进入正式正文
5. 每章完成后先做章节级审查，再继续下一章

### 9.4 收尾

1. 检查结构
2. 检查引用
3. 检查图表与代码片段路径
4. 更新 `CHANGELOG.md`
5. 冻结到对应版本目录

---

## 10. 交付前自检

交付前至少检查：

1. 是否遵守“先大纲、后正文”
2. 是否误覆盖旧版本
3. 图、表、截图、代码片段是否进入正确目录
4. 关键引用是否缺失
5. 是否把评价写成事实
6. 涉及“最新动态”的部分是否核对了日期
7. 是否遵守了“先研究、后写作、再审查”的章节流程
8. 当前章节是否仍停留在概述稿而未达到研究型正文初稿深度
