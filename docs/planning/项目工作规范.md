# 项目工作规范

## 1. 目的与优先阅读顺序

本文件是本仓库的长期工作手册，用于约束《具身智能行业和技术发展调研报告》的规划、写作、版本管理、资料沉淀与导出流程。

进入仓库后，默认按以下顺序阅读：

1. 本文件
2. `docs/planning/README.md`
3. `docs/planning/current/具身智能报告-当前详细大纲.md`
4. `docs/planning/current/当前版本修订计划.md`
5. `docs/planning/current/当前工作清单.md`
6. `CHANGELOG.md`
7. `docs/report/current/` 与最新冻结版本目录

---

## 2. `docs/planning/` 的职责

### 2.1 根目录只放长期通用规则

`docs/planning/` 根目录默认只保留两类文件：

1. 长期通用规范
2. 指向当前活跃规划与历史归档的导航文件

也就是说，根目录不再长期堆放只对某个版本阶段有意义的执行面板、临时推进方案或一次性导出设计稿。

### 2.2 `current/` 放当前活跃规划

`docs/planning/current/` 用于放与 `docs/report/current/` 对应的当前活跃规划文件，至少包括：

1. `具身智能报告-当前详细大纲.md`
2. `当前版本修订计划.md`
3. `当前工作清单.md`

其中：

1. “当前详细大纲”负责定义当前工作版的章节职责与结构边界
2. “当前版本修订计划”负责定义本轮版本迭代的目标、范围、问题清单与退出条件
3. “当前工作清单”负责定义下一批具体执行任务、对应章节位置、状态与批次推进顺序

### 2.3 `archive/vX.Y/` 放历史版本规划

`docs/planning/archive/vX.Y/` 用于归档该版本真正写作过程中使用过的版本性规划文档，例如：

1. 冻结时对应的大纲快照
2. 批次执行计划
3. 低打扰推进方案
4. 导出实现方案
5. 版本阶段性的精修计划

凡是只对某个版本或某个阶段有意义的文件，都应在版本冻结后归档到对应 `archive/vX.Y/`，而不是长期留在根目录。

### 2.4 冗余规划文件的处理原则

如果某份规划文档的内容已经被长期规范吸收，后续处理顺序应为：

1. 先把可长期复用的规则并入本文件或 `docs/planning/README.md`
2. 再把原始文档归档到对应版本目录
3. 不在根目录保留功能重复的平行说明文件

## 3. 版本启动与规划流程

### 3.1 新版本启动时必须先做规划

任意 `vX.Y` 版本启动时，默认先做以下动作：

1. 阅读上一冻结版本正文与 `CHANGELOG.md`
2. 检查 `docs/planning/current/` 中的现行大纲、现行修订计划与当前工作清单
3. 明确本次更新是“结构性迭代”还是“非结构性维护”
4. 更新 `docs/planning/current/` 对应文件
5. 再开始正文改写

### 3.1.1 新会话或新轮次开工前的强制预读

无论是新版本启动、同版本新轮次推进，还是重新打开一个新的聊天会话，都必须在正文动笔前完成以下预读：

1. `AGENTS.md`
2. `docs/planning/项目工作规范.md`
3. `docs/planning/README.md`
4. `docs/planning/current/具身智能报告-当前详细大纲.md`
5. `docs/planning/current/当前版本修订计划.md`
6. `docs/planning/current/当前工作清单.md`

其中：

1. `AGENTS.md` 负责在仓库级入口显式声明“Read First”与工作流规则
2. `项目工作规范.md` 负责定义长期通用机制
3. `当前详细大纲.md` 负责定义当前结构边界
4. `当前版本修订计划.md` 负责定义本轮修订目标
5. `当前工作清单.md` 负责定义本次真正要落笔推进的批次任务

未完成上述预读前，不得直接改写 `docs/report/current/` 正文。
进入正文批次后的第一条进度更新，还应显式说明“已完成预读”，以便在对话层面留下可见确认。

### 3.2 结构性迭代

下列工作属于结构性迭代：

1. 新增章节或删除章节
2. 调整章节职责或边界
3. 重命名章节并影响全书结构
4. 大幅改变章节内部组织方式
5. 大范围重写某章以改变论证框架

处理规则：

1. 必须先更新 `docs/planning/current/具身智能报告-当前详细大纲.md`
2. 必须等待用户明确确认该大纲
3. 若准备进入具体执行，还必须同步更新 `docs/planning/current/当前工作清单.md`
4. 只有确认后，才能继续写或重写正文

### 3.3 非结构性维护

下列工作属于非结构性维护：

1. 语言不通顺、术语不统一、句式不自然
2. 引用标签、链接融入方式、段尾来源格式不规范
3. 事实表述过满、推断与评价混写
4. 图表路径、交叉引用、版本说明等元信息修正
5. 导出脚本、版式脚本与可读性优化

处理规则：

1. 仍然必须先更新 `docs/planning/current/当前版本修订计划.md`
2. 每次开启新的正文批次前，还必须更新 `docs/planning/current/当前工作清单.md`
3. 若不改变全书结构，可不强制改动当前详细大纲正文层级
4. 只要用户已经明确提出该轮修订目标，即可在修订计划与工作清单更新后继续执行

### 3.3.1 工作清单的最小维护要求

`当前工作清单.md` 至少要写清：

1. 任务名称
2. 对应章节或章节区段
3. 具体补强目标
4. 当前状态（planned / in_progress / completed / blocked）

如果一次推进包含多条并行工作线，必须在工作清单中拆开，而不是只在对话里口头说明。

### 3.4 研究驱动回调大纲

当前详细大纲不是静态壳，而是研究骨架。

当章节级研究表明原结构会实质性损害正文质量时，允许重构，但必须遵守：

1. 先研究，后改大纲
2. 先改 `docs/planning/current/具身智能报告-当前详细大纲.md`
3. 再改受影响的 `docs/report/current/` 正文
4. 同步修正入口文件、交叉引用和图表编号说明

## 4. 报告正文与版本目录

### 4.1 正文目录

1. `docs/report/current/`：当前工作版
2. `docs/report/versions/vX.Y/`：冻结版本

### 4.2 工作原则

1. 平时只在 `docs/report/current/` 上持续推进
2. 冻结版本目录默认不直接改写
3. 新版本不覆盖旧版本，而是在 `current/` 基础上推进并在准备发布时新建 `versions/vX.Y/`
4. `00-*.md` 属于整书入口文件，应与当前版本号、当前大纲路径和当前状态保持一致

### 4.3 当前版与冻结版关系

1. `current/` 代表正在维护的最新版工作稿
2. `versions/vX.Y/` 代表某一时点冻结交付口径
3. 后续修订一律先进入 `current/`
4. 版本准备发布时，再把 `current/` 对应状态冻结到 `versions/vX.Y/`

## 5. 正文写作与修订流程

### 5.1 章节写作基本流程

每章默认遵守：

1. 明确本章要回答的问题
2. 做章节级研究与资料收集
3. 在 `research/notes/` 沉淀底稿
4. 写入 `docs/report/current/` 正文
5. 检查事实、推断、评价分层
6. 检查引用、图表位、公式位、代码片段位

### 5.2 章节最小交付标准

每章至少应尽量包含：

1. 核心问题与边界
2. 代表性路线或分析框架
3. 关键论文、系统、企业或产品案例
4. 工程实现路径与部署约束
5. 局限、争议与未解问题
6. 与前后章节的接口

### 5.3 低打扰推进

当用户授权连续推进时，可以按批次推进，而不是每章停一次。

但即使在低打扰模式下，也仍然必须遵守：

1. 先规划
2. 先研究
3. 再写正文
4. 再审查

## 6. 语言、引用与可读性规范

### 6.1 中文叙述为默认主体

正文默认以中文完成论述。英文只在以下情况保留：

1. 模型名、论文名、公司名、产品名、库名
2. 公认标准术语在首次出现时的原文
3. 公式、代码、接口字段与必要缩写

除了上述情况，不应把整段英文产品文案、整串英文功能枚举或英文营销语直接嵌进中文论述。

### 6.2 英文术语的推荐写法

推荐写法是：

1. 先给中文解释
2. 再在必要时保留英文原词或缩写
3. 后文以中文或中英混合的稳定术语继续引用

例如：

1. “协作机械臂（collaborative robotic arm）”
2. “共享自主（shared autonomy）”
3. “无 GPS 环境（GPS-denied environment）”

### 6.3 禁止“英文尾巴式”引用

禁止出现下列写法：

1. 一个中文段落末尾直接悬挂裸英文链接标签
2. 只剩一串英文来源名独立占行
3. 把官网标题原样塞进句尾，导致段落像突然断裂

正确做法应是把来源融入中文句子，例如：

1. “参见某公司官方产品页”
2. “该说法可在某论文的实验设置部分找到”
3. “相关能力描述见某官网说明”

### 6.4 事实、推断、评价分层

正文应尽量显式区分：

1. 事实：论文、官网、GitHub、报告、标准等可查证内容
2. 推断：基于多条事实做出的解释或归纳
3. 评价：作者判断、优先级排序、风险评估

不得把评价包装成事实，也不得把营销表述直接视为事实。

### 6.5 正文质量扫描

对长篇正文，建议在重要修订轮次至少执行一次全文扫描，重点检查：

1. 中英混杂密度过高的段落
2. 段尾裸链接或英文来源尾巴
3. 术语不统一
4. 只有一两句的薄弱三级标题
5. 事实与评价混写

仓库内可复用的正文扫描脚本应放在 `code/analysis/`。

## 7. 资料、研究卡与可复用资产

### 7.1 研究资料与正文分离

1. `research/` 放研究底稿、论文卡、企业卡、时间线与资料摘要
2. `docs/report/current/` 与 `docs/report/versions/` 只放整理后的正文与版本内资产
3. 不把临时搜索笔记直接塞入正文

### 7.2 可复用资产目录

1. `assets/images/`：可复用原图
2. `assets/diagrams/`：自绘流程图、结构图、关系图
3. `assets/screenshots/`：网页截图、产品页截图、公告截图
4. `docs/report/versions/<version>/figures/`：只属于某个冻结版的专用图
5. `docs/report/versions/<version>/snippets/`：只属于某个冻结版的专用代码片段
6. `docs/report/versions/<version>/tables/`：只属于某个冻结版的专用表格

### 7.3 企业卡与论文卡

长期跟踪类资产继续放在：

1. `research/companies/`
2. `research/papers/`

它们的职责是：

1. 为正文后续版本更新提供稳定研究接口
2. 为横向比较表和季度更新提供底层素材
3. 降低新会话重新起步的成本

### 7.4 结构化数据

1. `data/raw/`：未清洗原始数据
2. `data/processed/`：整理后的结构化结果

只有当资料已经稳定到适合复用时，才放进 `data/`，不要为了“目录完整”而空造数据文件。

## 8. 代码、导出与派生产物

### 8.1 `code/` 的职责

`code/` 主要用于：

1. 抓取脚本
2. 分析脚本
3. 表格生成脚本
4. 导出脚本
5. 质量检查脚本

### 8.2 导出脚本约定

报告导出入口统一放在 `code/export/`，默认复用以下链路：

1. `build_report_bundle.py`
2. `export_docx.py`
3. `export_pdf.py`
4. `run_export.py`

未来任意 `vX.Y` 版本都应继续复用同一套导出基础设施，而不是复制版本专属导出器。

### 8.3 导出产物定位

默认约定如下：

1. Markdown 是唯一可编辑源
2. Word (`.docx`) 是主要可读交付件
3. PDF 是辅助发布与打印件
4. 导出结果默认放到 `output/exports/<version>/`

### 8.4 导出文件不是编辑源

1. 不直接在导出的 `.docx` / `.pdf` 上进行长期版本维护
2. 如导出效果有问题，应优先修正 Markdown、图表资产或导出脚本
3. Word 和 PDF 只作为发布产物，不替代源稿

## 9. `tmp/`、缓存与 `.gitignore`

### 9.1 `tmp/` 的职责

仓库根目录下的 `tmp/` 视为本地临时实验区，用于：

1. 公式试排
2. 导出调试
3. 临时截图或探针文件
4. 不适合进入长期资料库的短生命周期中间物

### 9.2 `tmp/` 不是长期维护目录

`tmp/` 默认不是跨机器工作必需目录。即使在另一台电脑重新拉取仓库，也不应依赖 `tmp/` 中的内容才能继续项目工作。

因此：

1. `tmp/` 中的文件默认不应作为长期资产
2. `tmp/` 应加入 `.gitignore`
3. 若某个试验结果需要长期保留，应转存到 `research/`、`assets/`、`data/` 或 `docs/report/versions/<version>/`

### 9.3 其他应忽略的本地噪音

默认还应忽略：

1. `__pycache__/`
2. `*.pyc`
3. Word 临时锁文件，如 `~$*.docx`

## 10. skills 与质量门

### 10.1 本项目默认高频 skills

1. `writing-plans`
2. `brainstorming`
3. `academic-research-suite`
4. `paper-spine-audit`
5. `verification-before-completion`
6. `pdf`
7. `playwright`
8. `screenshot`
9. `jupyter-notebook`

### 10.2 默认调用顺序

章节级或批次级工作默认优先采用以下顺序：

1. 用 `writing-plans` 明确任务边界
2. 用 `academic-research-suite` 做研究
3. 用 `pdf` / `playwright` / `screenshot` / `web` 补证据
4. 写入 Markdown 正文
5. 用 `paper-spine-audit` 检查论证链
6. 用 `verification-before-completion` 做交付前自检

### 10.3 交付前最少检查项

在声明某轮工作完成前，至少检查：

1. 是否已更新当前版本修订计划或当前详细大纲
2. 是否误改冻结版本正文
3. 是否修正了对应的正文、入口文件和规范文件之间的链接
4. 是否完成关键章节的引用与术语一致性检查
5. 是否对新增脚本或大规模文本改动做了实际验证
