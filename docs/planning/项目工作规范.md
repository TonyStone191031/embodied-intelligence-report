# 项目工作规范

## 1. 目的

本文件是本项目的长期操作手册，供后续在新会话中继续维护《具身智能行业和技术发展调研报告》的大模型直接阅读。

进入本仓库后，优先阅读：

1. 本文件
2. `docs/planning/具身智能报告-V0.0-详细大纲.md`

---

## 2. 核心规则

### 2.1 先大纲，后正文

任何版本更新都必须先做大纲，不允许直接写正文。

标准顺序：

1. 阅读本文件与现有报告材料
2. 在 `docs/planning/` 新建或更新该版本大纲
3. 等待用户确认大纲
4. 大纲确认后，才开始正文写作

### 2.2 增量更新，不推倒重写

新版本默认基于旧版本增量修订：

1. 新论文、新模型、新企业、新政策补充
2. 旧判断修正
3. 图表、代码、引用和案例复用

### 2.3 区分三类内容

报告内容尽量分清：

1. 事实
2. 推断
3. 评价

不得把推断或评价写成事实。

## 3. 目录职责

### 3.1 `docs/planning/`

放规划文件：

1. 工作规范
2. 各版本详细大纲
3. skills 调用备忘
4. 其他阶段性规划文档

### 3.2 `docs/report/`

放报告正文与版本目录：

1. `current/`：当前正在维护的工作版本
2. `versions/`：已冻结的历史版本

### 3.3 `research/`

放调研过程资料，不放正式正文：

1. 论文笔记
2. 企业笔记
3. 数据集与 benchmark 说明
4. 行业文章摘录
5. 时间线材料

### 3.4 `assets/`

放图片资产原件与可复用图示：

1. `images/`
2. `diagrams/`
3. `screenshots/`

### 3.5 `code/`

放抓取、分析、制图脚本。

也用于放报告导出与编排脚本，例如：

1. 分章 Markdown 合并脚本
2. DOCX 导出脚本
3. PDF 导出脚本
4. 一键导出调度脚本

### 3.6 `data/`

放原始数据与处理后数据。

### 3.7 `output/`

放导出产物，如合并版 markdown、pdf、docx、截图输出。

其中长期建议固定保留：

1. `output/exports/<version>/`：某一版本的导出产物目录
2. 导出日志、检查报告与中间产物说明

## 4. 大纲与正文流程

### 4.1 新版本启动时

先检查：

1. `CHANGELOG.md`
2. 上一版本正文
3. 上一版本大纲
4. 本文件

### 4.2 新大纲放哪里

统一放在：

`docs/planning/`

命名建议：

`具身智能报告-VX.Y-详细大纲.md`

### 4.3 什么时候开始写正文

只有在用户明确确认大纲后，才允许写正文。

### 4.4 正文如何组织

正文按章节拆分维护，不长期维护单个超大 markdown。

### 4.5 正文默认对齐已确认大纲，但允许“研究驱动重构”

本项目默认要求正文对齐已确认详细大纲，但该大纲不是不可更改的静态壳，而是会随着深度研究推进而持续校准的研究骨架。

基本原则：

1. 在未出现明显结构性问题前，`docs/report/current/` 下的章节文件应与当前有效大纲中的“部分”一一对应
2. 每个章节文件内部的小节顺序，应与当前有效大纲中的该部分顺序保持一致
3. 正文写作时可以在当前有效大纲之下继续下探更细标题
4. 如果在调用 skills、补充文献和完成章节级研究后，发现现有大纲会实质性限制正文质量、论证完整性或章节边界合理性，则允许触发局部或阶段性重构
5. 发生重构时，必须先回到 `docs/planning/` 更新详细大纲，再继续受影响章节的正文写作
6. 对于已写章节，如其职责已因重构发生变化，应同步回修其标题、导语、结构说明和交叉引用，避免“旧结构正文”挂在“新结构大纲”下
7. 入口文件、图表编号、表格编号、代码片段编号与交叉引用，应尽量跟随最新有效大纲同步维护

重构触发条件：

1. 章节级深度研究表明原大纲对问题切分过浅、过窄或存在明显错位
2. 新增材料显示多个章节边界划分不合理
3. 当前章节若继续沿原大纲写作，会明显损害报告的完整性、严谨性或长期可维护性

重构流程：

1. 先做章节级研究
2. 明确原结构的问题
3. 在 `docs/planning/` 更新详细大纲
4. 向用户同步说明调整理由
5. 再继续相关正文编写或回修

禁止事项：

1. 未更新大纲就直接长期沿另一套结构写正文
2. 研究已证明结构失真，但仍机械坚持旧大纲推进
3. 正文文件名与当前有效大纲长期不一致
4. 在正文、图表和引用中并行维护多套编号体系，导致后续版本无法维护

### 4.6 正文必须按“研究型章节流程”推进

本项目的正文不是科普稿，也不是先写概述、后补证据的随手草稿。每一章都应按“先研究、后写作、再审查”的研究型流程推进。

章节级标准流程：

1. 先明确本章目标
2. 先做章节级研究与资料收集
3. 再写正式正文
4. 再做章节级审查
5. 通过后才继续下一章

其中：

1. “先明确本章目标”包括明确本章要回答的问题、章节内部结构、关键争议、图表需求、引用需求
2. “章节级研究与资料收集”包括论文、综述、技术报告、项目主页、GitHub、官方博客、企业发布材料、benchmark 页面等
3. “正式正文”必须建立在已整理的研究材料上，不允许先凭印象写概述再长期拖延补证据
4. “章节级审查”必须检查论证链、引用密度、事实/推断/评价分层、章节覆盖范围和图表规划是否达标

禁止事项：

1. 未形成该章研究材料就直接大段写正文
2. 用少量常识性描述替代系统研究
3. 章节只有概念介绍，没有代表路线、工程约束、局限性和争议分析
4. 把“后面再补引用”作为常态工作方式

### 4.7 章节级最小交付标准

每一章在被视为“正文初稿”前，至少应尽量包含以下要素：

1. 本章核心问题与边界
2. 该方向的基本定义与背景
3. 代表性技术路线或分析框架
4. 关键论文、项目、系统或企业实例
5. 工程实现路径或现实部署约束
6. 局限性、争议点与未解决问题
7. 后续章节接口，即本章与后文的衔接关系

如果某章暂时做不到上述全部内容，必须明确标注缺口，而不是默认以简短概述稿充当正式正文。

### 4.8 章节级推荐工作流与强制质量门

本项目推荐并优先采用如下章节工作流：

1. 用 `writing-plans` 设计该章任务拆解
2. 用 `academic-research-suite` 做章节级深度研究
3. 结合 `pdf`、`playwright`、`screenshot`、`web` 补论文、官网、GitHub、图示与截图证据
4. 在 `research/` 中沉淀研究材料与笔记
5. 再写入 `docs/report/current/` 对应章节正文
6. 用 `paper-spine-audit` 的思路检查论证链与章节深度
7. 用 `verification-before-completion` 的思路做交付前自检

其中第 2、5、6 步对本项目的研究型章节写作而言应视为默认强制环节，而不是可选优化项。

### 4.9 章节写作中的“动态回调大纲”机制

本项目采用“先有整体大纲，再做章节级重估”的工作方式。

即：

1. 先调用适合的 skills 制定一版全书整体大纲
2. 在编写每一章前，继续调用适合的 skills 做该章研究
3. 如果该章研究表明原大纲章节边界不合理，则允许回调并优化整体大纲
4. 大纲更新后，再同步该章及相邻章节的职责与接口

因此，大纲确认并不意味着后续不能调整；它意味着后续调整必须有研究依据、有文档同步，并对已写内容负责。

### 4.10 低打扰批次推进机制

当用户明确授权后，长篇报告写作可切换为“低打扰批次推进”模式，而不再要求每完成一章都单独等待用户确认。

该模式下：

1. 仍然必须坚持“先研究、后写作、再审查”
2. 章节级结构仍允许依据研究结果回调大纲
3. 但沟通与汇报单位从“单章”提升为“批次”
4. 每个批次结束后，再统一向用户汇报结构变化、章节状态和后续计划

默认批次定义、上下文管理策略、图表 / 公式 / 代码片段策略，见：

`docs/planning/V0.0-低打扰自主推进方案.md`

## 6. 报告格式要求

1. 报告使用 Markdown
2. 标题带版本号，例如 `《具身智能行业和技术发展调研报告-v0.0》`
3. 文风偏正式、严谨
4. 关键观点尽量带来源说明
5. 关键架构、时间线、模型谱系、对比关系尽量配图或配表
6. 关键方法必要时配代码片段或伪代码

每个技术方向尽量回答：

1. 是什么
2. 为什么重要
3. 怎么做
4. 工程上怎么落地
5. 局限是什么
6. 未来可能怎么发展

## 7. 资料组织规则

### 7.1 研究资料与正文分离

1. 正文文件只写整理后的结论和论述
2. 原始调研笔记放 `research/`
3. 抓取脚本和分析脚本放 `code/`

### 7.2 图片放置规则

1. 可复用原图：`assets/images/`
2. 自绘图：`assets/diagrams/`
3. 网页/公告截图：`assets/screenshots/`
4. 某版本专用图：`docs/report/versions/<version>/figures/`

### 7.3 代码片段放置规则

1. 分析脚本：`code/analysis/`
2. 抓取脚本：`code/scrapers/`
3. 某版本专用片段：`docs/report/versions/<version>/snippets/`
4. 报告导出脚本：`code/export/`

### 7.4 表格放置规则

1. 某版本整理后表格：`docs/report/versions/<version>/tables/`
2. 底层数据：`data/raw/` 或 `data/processed/`

### 7.5 命名建议

1. 论文笔记：`年份-机构-标题缩写.md`
2. 企业笔记：`企业名-更新日期.md`
3. 时间线：`timeline-主题.md`
4. 截图：`日期-来源域名-主题.png`
5. 图表：`章节号-主题.ext`

### 7.6 企业长期跟踪卡规范

`research/companies/` 中的“企业长期跟踪卡”不是正文替代物，而是后续企业章节、季度更新和跨公司比较的底层研究资产。后续无论新开哪个聊天窗口，大模型都应优先复用和更新这些卡片，而不是每次重新从新闻流开始。

创建或更新企业长期跟踪卡时，默认流程如下：

1. 先阅读对应章节正文与现有研究笔记，确认该企业在报告中的角色定位
2. 优先查证官网、技术博客、论文、GitHub、产品页与高可信采访，再补媒体或行业评论
3. 先更新 `research/companies/企业长期跟踪卡-模板-v0.0.md` 对应结构，再写具体企业卡
4. 若信息属于短期动态，例如融资、客户、事故、发布会、试点，应优先写入季度更新日志，而不是把长期卡写成新闻时间线
5. 企业卡更新后，如其判断会影响 `docs/report/current/20-22` 或 `23-24` 的口径，应同步回修相关章节或表格

企业长期跟踪卡建议至少包含：

1. 基本信息与当前主路线标签
2. 本体/平台与系统集成路线
3. 模型、数据与训练闭环路线
4. 制造、交付、客户与部署方式
5. 主要风险、证据缺口与容易被误读的点
6. 后续季度最值得跟踪的 3-5 个变量
7. 参考链接区，明确区分官网、论文、GitHub、访谈或新闻

命名建议：

1. 长期跟踪卡：`企业名-长期跟踪卡-v0.0.md`
2. 季度更新日志：`企业名-季度更新日志-v0.0.md`

默认优先建立长期跟踪卡的对象：

1. 海外：Google DeepMind、NVIDIA、Figure、Agility、Apptronik、Physical Intelligence
2. 国内：优必选、智元、宇树，以及后续正文中已形成稳定专题的公司

### 7.7 论文卡规范

`research/papers/` 中的“论文卡”用于沉淀代表性论文的稳定结构化理解，服务于 `11`、`12`、`18` 章以及后续版本更新。论文卡应帮助大模型快速回忆“这篇论文解决什么、接口是什么、在谱系中的位置是什么”，而不是只保留一段摘要。

创建或更新论文卡时，默认流程如下：

1. 先确认该论文所属主线，以及它在详细大纲中的对应章节
2. 优先阅读论文原文、项目主页、官方代码仓库与作者技术说明
3. 先按模板沉淀结构化信息，再决定是否回写正文
4. 如论文改变了原有谱系判断，应先同步 `docs/planning/具身智能报告-V0.0-详细大纲.md` 或相关章节职责，再修正文
5. 论文卡完成后，如适合进入对照表或时间线，应同步更新 `data/processed/` 中相应 csv

论文卡建议至少包含：

1. 问题设定与为什么重要
2. 输入输出接口、动作表示或规划接口
3. 数据来源、训练方式与评测方式
4. 相对前代工作的主要推进
5. 工程价值、部署含义与外推边界
6. 局限性、失败模式与后续影响
7. 参考链接区，至少包含论文链接，若有则补项目页和代码仓库

命名建议：

1. 单篇论文卡：`论文名-论文卡-v0.0.md`
2. 路线清单：`路线名-论文清单-v0.0.md`

默认优先建立论文卡的对象：

1. VLA 主线：RT-1、RT-2、PaLM-E、OpenVLA
2. 规划主线：SayCan、Code as Policies
3. 后续按正文需要继续扩展到世界模型、JEPA、Diffusion Policy、GR00T 等

### 7.8 研究卡与正文、表格之间的回写关系

研究卡不是孤立文件。后续任何一次大版本或季度更新，都应显式检查以下回写关系：

1. 企业长期跟踪卡更新后，是否需要回写 `20-企业分析框架.md`、`21-海外重点企业专题.md`、`22-国内重点企业专题.md`、`23-应用落地与商业化.md`、`24-产业格局、资本与政策.md`
2. 论文卡更新后，是否需要回写 `11-VLA 与机器人基础模型.md`、`12-规划、推理与决策的新路线.md`、`18-代表性论文与模型谱系梳理.md`
3. 若更新涉及可结构化字段，应优先更新 `data/processed/*.csv`，再通过脚本导出到 `docs/report/current/tables/`
4. 正文中引用研究卡时，应把研究卡视为内部研究资产，外部事实仍需优先指向论文、官网、GitHub 或官方报告链接

### 7.9 企业季度更新日志规范

`research/companies/` 中的“企业季度更新日志”用于承接高频变化信号。它的职责不是重写企业简介，而是把季度新增事件压缩成可回写的判断素材。

默认规则如下：

1. 长期稳定判断写入长期跟踪卡，季度变化写入季度更新日志
2. 季度更新日志优先记录融资、合作、发布、客户部署、事故、团队变化、政策与标准信号
3. 每条季度信号都应尽量标注“影响层”和“是否改变长期判断”
4. 若某条季度信号不足以改变长期判断，也应明确写“维持”，而不是默认沉默
5. 当季度日志累计到足以改变章节口径时，再回写长期跟踪卡、企业专题正文和 `data/processed/企业季度跟踪表-v0.0.csv`

命名建议：

1. 模板：`企业季度更新日志-模板-v0.0.md`
2. 实例：`企业名-季度更新日志-v0.0.md`

### 7.10 路线清单规范

`research/papers/` 中除单篇论文卡外，还应维护“路线清单”文件。路线清单的作用，是在单篇论文之上再提供一层“主线级入口”，便于后续季度增量更新时先判断主线结构，再决定是否补单篇卡或正文。

默认规则如下：

1. 路线清单按研究主线组织，而不是按年份罗列
2. 每份路线清单应说明用途、当前主线划分、更新规则和后续待补对象
3. 路线清单中优先链接到已有论文卡，而不是重复写长摘要
4. 当某条主线明显扩张到足以单独维护时，应新建新的路线清单，而不是持续把所有内容堆进 `18` 章正文

当前建议至少维护以下路线清单：

1. VLA
2. 世界模型
3. 规划与具身推理
4. 生成式动作与策略
5. 硬件、部署与系统工程
6. 仿真、评测与基础设施
7. 开源生态与工具链

## 8. 版本管理规则

### 8.1 不建议“每个版本一个 git 分支”

默认不用 git 分支来管理每个报告版本。

原因：

1. 分支适合并行开发和实验
2. 报告版本更适合做内容演进
3. 多分支会让图表、引用和章节历史变得分叉且难维护

### 8.2 采用“目录版本化 + Git 历史”

默认采用：

1. Git 主线维护整体历史
2. 报告版本通过文件与目录管理
3. 重大实验性改动才临时开分支

### 8.3 版本目录结构

```text
docs/report/
  current/
  versions/
    v0.0/
    v0.1/
```

含义：

1. `current/`：当前工作版本
2. `versions/vX.Y/`：已冻结版本

### 8.4 工作流

1. 平时在 `current/` 中工作
2. 某版本确认完成后，整理到 `versions/<version>/`
3. 新版本继续在 `current/` 上增量推进

### 8.5 避免覆盖旧版本

1. 已冻结版本目录默认不持续改写
2. 新版本不直接覆盖旧版本目录
3. 新版本必须新建自己的 `versions/<version>/`

### 8.6 版本资产管理

公共可复用资产继续放：

1. `assets/images/`
2. `assets/diagrams/`
3. `assets/screenshots/`
4. `code/analysis/`
5. `data/raw/`
6. `data/processed/`

版本专用资产冻结到：

1. `docs/report/versions/<version>/figures/`
2. `docs/report/versions/<version>/tables/`
3. `docs/report/versions/<version>/snippets/`

### 8.7 版本发布时同步更新

1. `CHANGELOG.md`
2. 对应版本目录
3. 版本入口文件

### 8.8 导出产物不是源文件

1. 项目的可编辑源文件以 Markdown 正文、版本目录内图表/代码片段、项目脚本为准。
2. 导出的 Word、PDF 属于发布产物，不作为后续版本演进的直接编辑源。
3. 默认以 Word (`.docx`) 作为主要可读交付稿，以 PDF 作为辅助发布/打印产物；若两者排版效果不一致，优先修正源 Markdown、图表资产或导出脚本，并先保证 Word 稳定可读。
4. 当本机环境具备 Microsoft Word 自动化能力时，PDF 应优先由已导出的 Word 产物再导出生成；只有在该能力不可用时，才回退到脚本直出 PDF。
5. 若发现目录、样式、分页、图注、交叉引用等问题，应优先修改源 Markdown、图表资产或导出脚本，然后重新导出。
6. 原则上不要在导出的 `.docx` 上手工修订后再把它回收为新的源文件。

### 8.9 固定化导出工作流

1. 日常写作继续维护分章 Markdown，路径为 `docs/report/current/`。
2. 当某一版本准备冻结时，先冻结正文到 `docs/report/versions/<version>/`。
3. 冻结完成后，再运行导出脚本生成合并稿、Word 与 PDF。
4. 导出结果默认放在 `output/exports/<version>/`，与正文源目录分离。
5. 推荐的导出脚本结构：
   - `code/export/build_report_bundle.py`
   - `code/export/export_docx.py`
   - `code/export/export_pdf.py`
   - `code/export/run_export.py`
6. 推荐职责划分：
   - `build_report_bundle.py`：按章节顺序合并 `docs/report/current/` 或 `docs/report/versions/<version>/`
   - `export_docx.py`：将合并稿导出为 Word，并统一标题、目录、页眉页脚、图表标题、代码块样式与正文段落排版
   - `export_pdf.py`：优先从已导出的 Word 产物生成 PDF；若当前环境无法调用 Word 自动化，再回退为从与 Word 同源的合并稿脚本直出 PDF
   - `run_export.py`：一键串联完整导出流程，例如 `run_export.py --version vX.X --source current|frozen`
7. 导出脚本应尽量做成可重复执行、可定位失败点的固定工具，而不是一次性临时脚本。
8. 数学公式优先以正文可读文本或结构化公式块维护，不应把 LaTeX 截图当作正式长期方案；导出脚本应同时支持 fenced `math` 与 `\[...\]` 两类块级公式写法。
9. 图表优先维护源 `mmd`、表格 Markdown 或结构化数据，不应直接把临时截图作为长期可编辑源。

### 8.10 版本收尾时的导出检查

1. 检查章节合并顺序是否与当前版本目录结构一致。
2. 检查 Markdown 中引用的图片、图表、代码片段在导出链路中是否可解析。
3. 检查 Word 与 PDF 是否成功落到 `output/exports/<version>/`。
4. 检查 Word 中的目录、正文、公式、流程图、表格与页眉页脚是否达到主要可读交付标准。
5. 检查 PDF 是否至少达到辅助发布标准，包括无明显路径泄露、公式未回退为原始 LaTeX、长表格未越界、主要图表可读；若 PDF 来自 Word 导出，还应确认其分页、目录层级与 Word 主稿保持一致。
6. 检查导出产物是否与冻结版本目录一一对应，避免把 `current/` 的未冻结内容误导出为正式版本。
7. 若导出失败，应记录失败原因属于素材缺失、路径错误、样式问题还是导出脚本缺陷。

## 9. 新版本标准流程

### 9.1 启动

1. 阅读本文件
2. 阅读上一版本正文
3. 阅读上一版本大纲
4. 阅读 `CHANGELOG.md`
5. 调研新增内容

### 9.2 规划

1. 新建该版本详细大纲
2. 标出新增、修订、删除或降级章节
3. 明确本版本重点增量
4. 等待用户确认

### 9.3 写作

1. 按章节推进
2. 先基础层，后技术层，再产业与结论层
3. 每章同步补图、表、引用和待核实项
4. 每章先完成研究材料，再进入正式正文
5. 每章完成后先做章节级审查，再继续下一章

### 9.4 收尾

1. 检查结构
2. 检查引用
3. 检查图表与代码片段路径
4. 更新 `CHANGELOG.md`
5. 冻结到对应版本目录

## 10. 交付前自检

交付前至少检查：

1. 是否遵守“先大纲、后正文”
2. 是否误覆盖旧版本
3. 图、表、截图、代码片段是否进入正确目录
4. 关键引用是否缺失
5. 是否把评价写成事实
6. 涉及“最新动态”的部分是否核对了日期
7. 是否遵守了“先研究、后写作、再审查”的章节流程
8. 当前章节是否仍停留在概述稿而未达到研究型正文初稿深度

## 11. 目录启用状态补充说明

### 11.1 哪些目录当前为空是正常的

下列目录在项目早期或纯正文撰写阶段可以暂时为空：

1. `code/`
2. `data/`
3. `output/`

它们不是废弃目录，而是为后续长期维护预留的“按需启用型基础设施目录”。

### 11.2 `research/` 不是空目录

`research/` 用于沉淀研究底稿、论文笔记、企业笔记、时间线和资料摘要。
如果该目录中已经存在研究底稿，则说明它已处于启用状态，不应视为无用目录。

### 11.3 `code/` 在什么情况下启用

`code/` 只在形成可复用脚本时启用，而不是为了让目录“看起来完整”而预先塞内容。

典型启用场景包括：

1. 批量抓取论文、GitHub、企业官网或新闻页面元数据
2. 生成图表、时间线、统计表或章节辅助分析结果
3. 清洗和整理结构化资料
4. 导出合并版报告、辅助校验引用或检查链接

如果当前工作主要是人工研究、阅读和直接写 Markdown，`code/` 暂时为空是正常的。

### 11.4 `data/` 在什么情况下启用

`data/` 只在项目中已经形成可复用的结构化底层资料时启用。

典型启用场景包括：

1. 论文清单与元数据表
2. 企业跟踪表、版本对比表、时间线表
3. 图表背后的原始统计数据
4. 从外部资料导出的原始 CSV / JSON / XLSX

其中：

1. `data/raw/` 保存未清洗原始导出物
2. `data/processed/` 保存整理后的结构化结果

如果当前还没有稳定的数据资产，`data/` 暂时为空是正常的。

### 11.5 `output/` 在什么情况下启用

`output/` 是派生产物目录，不是正文源文件目录。

典型启用场景包括：
1. 当开始产出自动检查结果时。
2. 当开始批量生成图表、合并稿、统计清单或导出结果时。
3. 当需要保存“可再生成但不宜混入源目录”的中间产物时。
4. 当某一报告版本进入冻结与发布阶段时，`output/exports/<version>/` 应作为 Word / PDF 等发布产物的默认落盘位置。
5. 导出整书合并版 Markdown。
6. 导出 PDF / DOCX。
7. 导出分享用截图包。
8. 导出给外部工具或演示使用的中间产物。

如果当前仍处于研究与正文精修阶段，尚未开始集中导出，`output/` 暂时为空是正常的。

### 11.6 是否删除空目录

当前默认结论：

1. 保留 `code/`、`data/`、`output/` 目录
2. 将其视为长期项目型工作所需的预留目录
3. 不因为当前阶段暂时为空就删除

只有在后续较长阶段内确认项目完全不会沉淀脚本、结构化数据或导出产物时，才考虑删除，并同步回修本规范中的目录结构说明。
