# 项目工作规范

## 1. 文件定位

本文档是本项目的长期工作规范，面向后续在本仓库中继续维护《具身智能行业和技术发展调研报告》的大模型/代理。

作用有两点：

1. 让新会话中的大模型快速理解本项目的组织方式和更新规则。
2. 让报告版本演进、资料沉淀、引用管理、图表管理保持一致，不因上下文切换而失控。

建议任何进入本仓库继续工作的模型，先阅读本文件，再阅读：

1. `docs/planning/具身智能报告-V0.0-准备方案.md`
2. `docs/planning/具身智能报告-V0.0-详细大纲.md`

---

## 2. 工作总原则

### 2.1 先规划，后写作

任何新版本报告开始前，都要先做规划，不允许直接动笔写正文。

标准顺序：

1. 阅读本项目规范与现有版本材料。
2. 在 `docs/planning/` 下新建或更新该版本的大纲文件。
3. 等待用户确认大纲。
4. 只有在大纲确认后，才能开始正文写作。

### 2.2 优先增量更新，而不是推倒重来

新版本应优先基于旧版本的结构与资产做增量更新，包括：

1. 新论文、新模型、新企业、新政策的补充。
2. 对旧判断的修正。
3. 图表、代码片段、案例与引用库的复用。

### 2.3 事实、推断、评价分层

报告中的内容尽量区分三层：

1. 事实：论文、官网、GitHub、官方演讲、法规、标准等可追溯信息。
2. 推断：从多个事实归纳出的趋势判断。
3. 评价：对技术路线、企业策略、商业模式的分析性判断。

不得把推断或评价伪装成事实陈述。

### 3.1 `docs/planning/`

用于存放规划类文件，包括：

1. 准备方案
2. 详细大纲
3. 工作规范
4. skills 调用备忘
5. 后续版本的大纲或章节规划文件

### 3.2 `docs/report/`

用于存放报告正文相关文件。

建议采用：

1. 总入口文件
2. 按章节拆分的正文文件
3. 每个版本独立目录

### 3.3 `research/`

用于存放研究过程资料，不是最终报告正文。

适合放：

1. 论文阅读笔记
2. 企业档案
3. 数据集与 benchmark 说明
4. 行业文章摘录
5. 时间线材料

### 3.4 `assets/`

用于存放图像资产原件和可复用图示。

### 3.5 `code/`

用于存放抓取、分析、图表生成等代码。

### 3.6 `data/`

用于存放原始数据与清洗后数据。

### 3.7 `output/`

用于存放导出物，例如合并版 markdown、pdf、docx、截图产物等。

## 4. 大纲与正文的工作流程

### 4.1 新版本启动时必须先做什么

新版本启动时必须先：

1. 检查现有 `CHANGELOG.md`
2. 检查上一个版本的正文与大纲
3. 检查本规范与 skills 备忘
4. 新建本版本的大纲文件

### 4.2 大纲文件应该放在哪里

新版本大纲优先放在：

`docs/planning/`

命名建议：

`具身智能报告-VX.Y-详细大纲.md`

例如：

`具身智能报告-V0.1-详细大纲.md`

### 4.3 什么时候才能开始写报告

只有在用户明确确认大纲后，才能开始正文写作。

如果用户没有明确确认，应停留在规划阶段，不进入正文生成。

### 4.4 正文写作时如何组织文件

正文应继续采用拆章维护方式。

建议：

1. 总入口文件负责版本说明、章节索引、导出说明。
2. 正文章节分文件维护。
3. 图表、代码片段、表格尽量与版本目录配套管理。

## 6. 报告格式要求

### 6.1 基本格式

1. 报告使用 Markdown 编写。
2. 标题中要带版本号，例如 `《具身智能行业和技术发展调研报告-v0.0》`。
3. 表述偏正式、严谨。
4. 关键观点要带来源说明。

### 6.2 内容风格要求

1. 尽量避免空泛口号式表达。
2. 强调技术、工程、产业三线并行。
3. 每个技术方向尽量回答六类问题：
   - 是什么
   - 为什么重要
   - 怎么做
   - 工程上怎么落地
   - 局限是什么
   - 未来可能怎么发展

### 6.3 图、表、代码要求

1. 关键架构、时间线、模型谱系、系统框图建议配图。
2. 关键对比建议配表格。
3. 关键方法如有代表性代码接口或伪代码，可配代码片段。
4. 图、表、代码片段必须有清晰来源或自绘说明。

### 6.4 引用要求

1. 学术观点优先引用论文、项目主页、官方技术报告。
2. 企业与产品信息优先引用官网、GitHub、官方博客、发布会材料。
3. 行业趋势与市场数据应注明是媒体/研究机构/券商/咨询机构观点。

## 7. 资料组织规范

### 7.1 研究资料与正文分离

不要把研究过程材料直接塞进正文目录。

原则：

1. 正文文件只写组织后的结论和论述。
2. 原始调研笔记进入 `research/`。
3. 抓取脚本与分析脚本进入 `code/`。

### 7.2 图片组织规范

#### 可复用原始图片

放在：

`assets/images/`

#### 自绘架构图、流程图、示意图

放在：

`assets/diagrams/`

#### 网页截图、公告截图、发布会截图

放在：

`assets/screenshots/`

#### 某一报告版本专用图

放在该版本目录下的：

`docs/report/versions/<version>/figures/`

### 7.3 代码片段组织规范

如果是为报告生成图表、表格、统计结果的脚本，放到：

`code/analysis/`

如果是抓取信息的脚本，放到：

`code/scrapers/`

如果是某一报告版本专用的引用代码片段导出，可同步放到：

`docs/report/versions/<version>/snippets/`

### 7.4 表格组织规范

报告中使用的整理后表格，优先放到：

`docs/report/versions/<version>/tables/`

如果表格依赖底层数据，源数据要继续存放到：

`data/raw/` 或 `data/processed/`

### 7.5 文件命名建议

1. 论文笔记：`年份-机构-标题缩写.md`
2. 企业笔记：`企业名-更新日期.md`
3. 时间线：`timeline-主题.md`
4. 截图：`日期-来源域名-主题.png`
5. 图表：`章节号-主题.ext`

## 8. 报告版本管理规范

这是本规范中最重要的部分之一。

### 8.1 不建议把“每个报告版本”做成一个 git 分支

默认不建议用“每个版本一个 git 分支”的方式管理报告版本。

原因：

1. 分支适合并行开发、实验性修改、特性隔离。
2. 报告版本是内容演进，不是彼此孤立的软件功能。
3. 如果每个版本都单独开分支，长期会让图表、引用、章节修订在多个分支间分叉，后续合并和追溯会非常痛苦。
4. 这会让“最新主线版本”不清晰。

### 8.2 推荐采用“目录级版本管理 + Git 历史追踪”

默认采用：

1. Git 主分支维护整个项目历史。
2. 报告版本通过目录和文件命名管理。
3. 重大结构变动可临时开分支，但不是按版本常态化开分支。

### 8.3 推荐的版本目录结构

建议在 `docs/report/` 下新增：

```text
docs/report/
  current/
  versions/
    v0.0/
    v0.1/
    v0.2/
```

含义：

1. `current/`
   - 当前正在维护的工作版本
2. `versions/v0.0/`
   - 已冻结的 v0.0 版本内容
3. `versions/v0.1/`
   - 已冻结的 v0.1 版本内容

### 8.4 `current/` 与 `versions/` 的关系

建议流程：

1. 新版本开始时，在 `current/` 下组织工作稿。
2. 版本确认完成后，将该版本的稳定内容整理并固化到 `versions/<version>/`。
3. 下一个版本继续在 `current/` 基于上一版本增量演进。

### 8.5 如何避免覆盖老版本正文

规则：

1. 已冻结版本目录只做勘误，不做持续改写。
2. 新版本不直接覆盖旧版本目录。
3. 新版本目录必须新建，例如 `versions/v0.1/`。

### 8.6 如何避免覆盖老版本引用的图片、表格、代码片段

建议采用“双层资产策略”：

#### 公共可复用资产

继续放在：

1. `assets/images/`
2. `assets/diagrams/`
3. `assets/screenshots/`
4. `code/analysis/`
5. `data/raw/`
6. `data/processed/`

#### 某个版本冻结时的版本专用资产

复制或整理到：

1. `docs/report/versions/<version>/figures/`
2. `docs/report/versions/<version>/tables/`
3. `docs/report/versions/<version>/snippets/`

这样做的好处：

1. 冻结版本可以独立回看，不受后续资产改动影响。
2. 公共资产仍然可以在新版本中继续复用。
3. 不需要依赖分支才能找回某一版对应图表和片段。

### 8.7 报告文件命名规则

建议：

1. 总入口：`00-具身智能行业和技术发展调研报告-vX.Y.md`
2. 分章节：`01-...md`、`02-...md`
3. 导出物：放到 `output/exports/`

### 8.8 版本发布时应同步更新什么

每次新版本冻结时，应同步更新：

1. `CHANGELOG.md`
2. 对应版本目录
3. 版本入口文件
4. 版本专用图表/代码片段目录

## 9. 新版本报告的推荐工作流

### 9.1 新版本启动

1. 阅读本规范
2. 阅读上个版本正文
3. 阅读上个版本大纲
4. 阅读 `CHANGELOG.md`
5. 调研新增内容

### 9.2 规划阶段

1. 在 `docs/planning/` 新建该版本详细大纲
2. 标出新增章节、修订章节、删除或降级章节
3. 明确本版本重点增量
4. 等待用户确认

### 9.3 正文阶段

1. 优先按章节写作
2. 先基础层，后技术层，再产业层和结论层
3. 每章同步补图表、引用、注释与待核实项

### 9.4 收尾阶段

1. 做结构与引用检查
2. 做图表和代码片段目录检查
3. 更新 `CHANGELOG.md`
4. 冻结到对应版本目录

## 10. 其他必要规范

### 10.1 不要把用户尚未确认的结构直接固化成正文

如果结构性变化较大，应先更新大纲和规划文档，再改正文。

### 10.2 企业与论文章节要避免“列表化堆砌”

应尽量用统一分析框架组织，而不是简单罗列。

### 10.3 对最新动态必须做时间核查

凡是涉及：

1. 最新论文
2. 最新产品
3. 最新融资
4. 最新政策
5. 最新 benchmark

都应尽量核查具体日期。

### 10.4 对模型输出的自检要求

交付前至少检查：

1. 是否越过了“先大纲、后正文”的规则
2. 是否误覆盖旧版本
3. 图表、代码片段、截图是否进入正确目录
4. 引用是否缺失
5. 是否把评价写成事实
