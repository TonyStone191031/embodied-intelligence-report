# 当前版本修订计划

## 当前轮次

- 目标版本：`v0.2`
- 轮次性质：在 `v0.1` 冻结基线之上的“技术主干深度 + 工程案例 + 图表资产”迭代
- 对应正文目录：`docs/report/current/`
- 上一冻结版本：`docs/report/versions/v0.1/`

## 启动基线

1. `v0.1` 已冻结为可追溯 Markdown 源稿快照；其规划文件已归档到 `docs/planning/archive/v0.1/`。
2. `v0.2` 暂不改变全书部分、章节文件与既有编号体系；若研究表明章节职责必须调整，仍须先回写当前详细大纲并取得确认。
3. 本轮不把 Word/PDF 重新导出作为正文深化的前置条件。Markdown、研究底稿和可复用图表源文件继续是唯一可编辑源。

## 本轮目标

1. 对技术主干章开展“方法解释 + 工程案例 + 失败模式 + 可复用图表”的一体化补强，避免只增加评述性文字。
2. 将案例写成可复核的系统剖面：任务边界、输入输出、控制/学习接口、数据与运行时约束、失败恢复、可迁移结论。
3. 将图表从正文中的说明性占位提升为可复用资产：每张图必须有源文件、标题、正文锚点、服务的问题和必要的引用说明。
4. 继续将正文结论沉淀到 `research/notes/`、`research/papers/`、`research/companies/` 或 `data/processed/`，避免后续更新重新从零检索。

## 第一批优先范围

### 后续两批范围

1. 案例深化：`11 / 15 / 21 / 22 / 23 / 27`，分别覆盖 VLA、部署系统、企业路线、应用单元和长期兴趣专题的证据链。
2. 图表深化：优先服务 `05 / 06 / 08 / 13`；需要时再扩展到 `11 / 15 / 21 / 23 / 27`。先确认正文需要，再新增资产，避免只为凑数量绘图。

## 质量与退出条件

1. 每个目标章至少新增一段完整案例剖析，并写明其不能证明什么。
2. 每个新增图表都能从正文定位到服务的问题、源文件和引用/数据来源；图表不以营销素材替代技术论证。
3. 新增正文经过 `code/analysis/scan_prose_quality.py` 复扫，且不重新引入高风险中英混写、英文尾链或引用悬挂。
4. 每个批次同步形成研究底稿与变更记录；批次完成后才决定下一批是否需要大纲回调。
