# 第二十五部分 关键争议与作者判断框架

到这一部分，报告不再只是罗列材料，而必须显式收束为一套判断体系。因为具身智能的很多分歧，不是简单的对错，而是不同约束条件下优先级不同。本部分的目标，是把这些争议明确写出来，并给出一套之后可持续更新的个人判断框架。

## 112. 技术路线争议

### 112.1 端到端还是模块化

端到端路线更统一、更可能提升迁移；模块化路线更可控、更易验证。现实系统很可能长期位于两者之间的混合带。

### 112.2 通用基础模型还是垂类模型

通用模型有规模收益和复用价值；垂类模型更容易在高价值场景里快速落地。

### 112.3 世界模型是否会成为主线

世界模型很重要，但其是否会成为真正主线，仍取决于它能否在真实控制与部署链路中提供持续收益。

## 113. 工程路线争议

### 113.1 大脑和小脑应否解耦

从系统稳定性角度看，解耦几乎不可避免；从统一学习角度看，耦合又更诱人。

### 113.2 端侧与云侧如何分工

部署约束决定端侧必须承担更多责任，但训练和记忆系统又很难完全离开云侧。

### 113.3 数据优先还是模型优先

在机器人里，数据质量和闭环常常比盲目放大模型更关键；但没有足够统一的模型，数据也难充分复用。

## 114. 产业路线争议

### 114.1 人形是不是最佳商业形态

人形可能是最强通用平台想象，但未必是最近的商业最优形态。

### 114.2 哪些场景会最先规模化

当前更现实的仍是制造、物流、巡检与高约束服务子场景。

### 114.3 哪些公司最可能建立护城河

最可能建立长期护城河的，往往不是单次模型发布最亮眼的公司，而是同时掌握数据、本体、基础设施和部署反馈闭环的公司。

## 115. 作者判断框架

### 115.1 观察指标

建议持续跟踪：

1. 真实部署案例而非单次 demo
2. 失败恢复能力而非只看成功片段
3. 数据闭环与更新节奏
4. 本体成本与制造进展
5. 端侧部署与时延控制

### 115.2 应定期跟踪的论文、企业、产品、政策信号

论文看接口变化，企业看交付变化，政策看场景开放与监管边界变化。

### 115.3 如何避免被短期热点带偏

最有效的方法不是拒绝热点，而是把所有热点重新压回同一套分析坐标：它改变了哪一层？解决了什么硬问题？绕开了哪些问题？又留下了哪些未解之处？

本部分的最终结论是：具身智能最需要的不是情绪化站队，而是持续的结构化怀疑。好的判断框架，应该既允许对技术前景保持开放，又不断把每条新叙事压回物理、数据、部署和商业化现实中检验。
